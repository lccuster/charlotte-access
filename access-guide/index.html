Liam's Reference Sheet to Digital Accessibility

This page is designed to be a supplemental material for those created by any other UNC Charlotte entities, with the express intent of making something directly applicable to the situation of educators on campus, most specifically the College of Computing and Informatics. As such, I've broken this guide down by a few general types of things I think we as educators need to be expressly concerned with in making our educational materials accessible.

Interpreting Web Content Accessibility Guidelines (WCAG) is a little difficult, so I've relied on an external resource which has created a wonderful downloadable checklist for WCAG compliance so that I don't have to. I've taken the extra step of organizing this content down by type of material, or course consideration, which needs to be taken, such that we can focus on immediate needs for each type of work item.

If I missed anything, you have suggestions, or have found a helpful resource (or solution to some problem!), please feel free to reach out to me at my email lcuster@charlotte.edu, and I'm more than happy to update or add stuff. Again, this is intended to be a helpful resource for everyone as we try to meet the requirements of Title II.

Interpreting This Guide

As you work to make your course accessible, you need to consider every element of the course and how you need to make adjustments (if you do) to be accessible. This guide provides basic definitions of a few types of course content (webpages, documents, etc.) and then a list of general accessibility tasks, along with specific WCAG codes, to reach accessibility compliance with that component.

Of great importance: A single piece of course content can and most likely will be classified in multiple categories. For example, a pdf file detailing an assignment is both an assignment AND a document. Generally speaking, almost everything is a document, and other tags will apply based on their definitions and your interpretation. When in doubt, subject what you're doing to more guidelines, rather than less.

Course Content

Assignments

Description

This section shows changes required for any assignment given in the course; this could be through an LMS, website, on paper, verbally, anything. If it is a task a student will do, it should consider these requirements.

Sensory Instructions

Specific Description

WCAG 2.0A Criterion 1.3.3: When providing detailed instructions, make it so they aren't reliant on a single sensory ability.

Tips to Comply

When creating instructions for assignments or lessons, make sure that you don't rely on a single sensory ability. A lesson centered around "looking" at something clearly needs alternative instructions, or alternative design, to accommodate those that can't see. You can extrapolate this to hearing, touch, tasting, and smelling too.

Documents

Description

Documents includes anything that students need to parse. This is Word files, pdfs, websites, LMS pages, anything that involves a handout or piece of information students interact with.

Alt Text

Specific Description

WCAG 2.0A Criterion 1.1.1: All images and non-text content that conveys necessary meaning or information needs a text alternative.
WCAG 2.0AA Criterion 1.4.5: Do not use images of text unless necessary (e.g., logo).

Tips To Comply

When creating alt text for an image, consider what you want readers to get from that image. A good alt text description is not overly long, and focuses on just the core idea that needs to be conveyed from the image; do not include superfluous information, like the colors of clothing, adjectives describing things, and similar "fluff," unless they are necessary to convey. As part of that, expecting users to "read" an image isn't great; just don't do it.

Additionally, if there isn't anything important to convey in an image, mark it as decorative. In raw html, this means leaving the alt text as an empty string ("").

Color

Specific Description

WCAG 2.0A Criterion 1.4.1: Do not rely on color alone to convey information.
WCAG 2.0AA Criterion 1.4.3: There must be a color contrast ratio of at least 4.5:1 between regular text and background and 3:1 for large text.

Tips to Comply

These two criterion ask you to carefully consider how color is used in your documents; the first specifies that you need more than color to signify the difference between items in your documents. Other ways to convey information in this way is texture (for images, graphs), typeface (for text) or tagging (such as headers, text tags). The second asks you to use high-contrast color combinations for content that is visually important, to help break them out from the background. White on black is fine, but for everything else, use this color contrast checker.

Hyperlinks

Specific Description

WCAG 2.0A Criterion 2.4.4: The purpose of each link should be clear based on its anchor text (e.g., don't use “click here”, etc.).

Tips to Comply

This is an important one, and takes time and skill to develop! When a user using a screenreader looks over a page, just like when we press tab, they can quickly access all links on a page, and their screenreader will tell them the hyperlink text. As a sighted person, I rely on the context around the "click here!" to figure out where it will take me; cut it out! Make the hyperlink text itself describe where the link takes the user (i.e. "KhanAcademy website" or "fully online course textbook")

Reading Order

Specific Description

WCAG 2.0A Criterion 1.3.2: If sequence of content affects meaning, a correct reading order can be programmatically determined.

Tips to Comply

As this applies to any document, it means that there needs to be some way for a user accessing the document through technology (for example, a screenreader) should be able to experience the correct reading order. A fantastic example of where this can fall apart is two-column, where without proper tagging, there is no established reading order, so content is read left to right, regardless of column.

If you're making a document format (pdf, pptx, etc), ensure that the reading order is set correctly; this can often be done by using an accessibility checker in your software to make it or edit it. It is also worth noting that source documents (docx, odf, etc) are far easier to access than rendered formats (pdf). If you want to check the reading order for your documents, check out Virginia Tech's Reading Order Resources.

If you're making a website, make sure that the content is laid out in the source code in your intended reading order; this ensures that those accessing it programmatically will get the experience you intend.

Lectures

Description

This section is not just for online lectures or recordings, which are subject to these criterion, but also all lectures generally. Practicing accessible presenting skills will make the work creating or refining digital lecture material far easier, since you'll potentially have practice (and recordings!).

Audio Descriptions

Specific Description

WCAG 2.0A Criterion 1.2.3: For any video where meaningful information is not available audibly, add a transcript or an audio description describing that information.
WCAG 2.0AA Criterion 1.2.5: An audio description is optional under 1.2.3 level A but not in 1.2.5 AA. The best route is to provide both a transcript (optional) and an audio description but the audio description is required.

Tips to Comply

Giving effective audio descriptions is a skill, and one that may take time to develop. The core idea is to stop relying on visual cues or instructions in your lecture content, and instead sharing the information that would be gained visually through audio. This means creating a sort of "alt text" for images in your lectures, but it also means relying less on visual cues to support your lecture. Try removing phrases like "if you look here..." from your vocabulary when teaching, since to a VI student, this means nothing.

Live Captions

Specific Description

WCAG 2.0AA Criterion 1.2.4: More formal, live presentations must have closed captions.

Tips to Comply

When presenting in class, it is best practice to have captions, whether you are recording or not. If you utilize some slides (powerpoint, google slides), both of those tools have built in capacity to generate live captions, you just need to look up how for your program.

If you are switching between content, or relying on content outside of a slide deck, then the next best option it to utilize a program like Zoom to provide automatic captions for your class. If your class is in person, utilize the second monitor provided in classrooms to share content from one screen to the other; make sure that Zoom is the available screen being shared, so that students can see the captions. Finally, while you're here, why not provide hybrid flexibility, and also record your lecture for the future? Most of the steps have already been taken.

Videos and Audio

Description

This refers to all non-text, non-document based media given to students that isn't interactive: think of recordings, podcasts, audiobooks, etc. As with everything else, ADA Title II applies not just to the content you make, but the content you decide to share as well. Make sure that all audio and videos you share meet these goals.

Video and Audio Alternatives

Specific Description

WCAG 2.0A Criterion 1.2.1: All video-only and audio-only content has a text transcript. Transcripts are clearly labeled and available near the media.
WCAG 2.0A Criterion 1.2.2: All video with sound has closed captioning.
WCAG 2.0A Criterion 1.2.3: For any video where meaningful information is not available audibly, add a transcript or an audio description describing that information.
WCAG 2.0AA Criterion 1.2.5: An audio description is optional under 1.2.3 level A but not in 1.2.5 AA. The best route is to provide both a transcript (optional) and an audio description but the audio description is required.

Tips To Comply

These four compliance goals focus on creating accessible video and audio content for students. There are a few key goals in this, which you should always aim for:

  1. Always have a text transcript available for visual/audio content.
  2. When creating videos, make sure that they still convey all content with audio only.
  3. When creating videos, make sure that they still convey all content without any audio.

The first goal is relatively easy to reach, especially for videos uploaded to Canvas Studio; it automatically generates captions, but YOU MUST review and correct those captions before distributing it. One those captions are generated, you can also export and use them as a transcript alternative for students.

As far as creating meaningful alternatives, criterion 1.2.3 and 1.2.5 essentially ask you to make sure that listeners without visuals can get the entire experience. A great way to check this is by listening to any video you create or use without the visuals, like a podcast. If you cannot fully understand it (or rather, a student couldn't fully understand it) from just the audio alone, it's not compliant. Criterion 1.2.2 also asks for the opposite, so a good way to check this for access is to watch videos on mute, and rely only on the captions. Again, determine if this method of viewing also effectively conveys content.

Webpages

Description

This section details requirements for webpages generally. This includes LMS pages, but also your own site(s) you may link to, and any website you link to, even if you don't own or run it in anyway.

Consistency and Labeling

Specific Description

WCAG 2.0AA Criterion 3.2.4: Components that have the same function within a website are identified consistently (but not necessarily identically).
WCAG 2.0A Criterion 3.1.2: Provide clear visual and programmatic labels or instructions for user input fields, ensuring users understand what information is required and how to input it.
WCAG 2.0A Criterion 4.1.2: For all user interface components (including forms, links, components generated by scripts), the name, role, state, and/or value should all be able to be programmatically determined.

Tips to Comply

Everything should be labeled, and you should be consistent about how you label it. Enough said.

Error Resolution

Specific Description

WCAG 2.0A Criterion 3.3.1: Make any form errors easy to identify, understand, and correct.
WCAG 2.0AA Criterion 3.3.3: If an input error is automatically detected, then suggestions for correcting the error should be provided.
WCAG 2.0AA Criterion 3.3.4: For pages that create legal commitments or financial transactions or any other important data submissions, one of the following is true: 1) submissions are reversible, 2) the user has an opportunity to correct errors, and 3) confirmation is available that allows an opportunity to review and correct before submission.

Tips to Comply

When asking users for input, it is best practice to make it clear to them where errors arise, and when possible, prevent them from submitting erroneous content. This isn't super applicable for course content, but when it is, try and make it clear how we can actually overcome these issues!

Flashing Elements

Specific Description

WCAG 2.0A Criterion 2.3.1: Web pages do not contain anything that flashes more than three times in any one second period.

Tips to Comply

Don't make using your website painful and/or harmful.

Font Sizing

Specific Description

WCAG 2.0AA Criterion 1.4.4: Text must be able to be resized up to 200% without negatively affecting the ability to read content or use functions.

Tips to Comply

Make sure that your page scales well when text is scaled up (elements don't break, get truncated, etc), so that users that need to magnify the text can have a clean experience.

Keyobard Navgiation

Specific Description

WCAG 2.0A Criterion 2.1.1: All content and functions on a website must be accessible by keyboard only (i.e., no mouse).
WCAG 2.0A Criterion 2.1.2: Keyboard-only users must never get stuck on any part of the website; they must be able to navigate forwards and backwards.
WCAG 2.0A Criterion 2.4.1: Provide a way to bypass the header and go straight to the main content such as a skip link. Landmarks and headings also work.
WCAG 2.0A Criterion 2.4.3: Users must be able to navigate through a website in a logical, sequential order that preserves meaning.
WCAG 2.0AA Criterion 2.4.7: Any “user interface control” that receives focus from a keyboard user should have a focus indicator that indicates that focus visually.
WCAG 2.0A Criterion 3.2.1: Nothing changes merely because an item receives focus; a user must actively choose to activate an item (e.g., hit enter to submit) before a change takes place.
WCAG 2.0A Criterion 3.2.2: User interface changes or actions shouldn't automatically occur due to input (e.g., form doesn't auto submit once all fields are filled out).

Tips to Comply

The down and dirty here is to make your website keyboard accessible, which is something anyone can test. If nagivating your pages is cumbersome or annoying, its likely not accessible. Of particular note here is the importance of focus, which is the state an element will be in when hovered or tab-selected; make sure to let users know where they are, and don't make focus do something that users may not want (like opening a dropdown of options, for example). In general, let users know where they are, make navigation easy and make sense, and don't do anything without the user initiating it by pressing a button on their keyboard.

Language Identification

Specific Description

WCAG 2.0A Criterion 3.1.1: Set the language for your website.
WCAG 2.0AA Criterion 3.1.2: Indicate any language changes for an entire page or within the content.

Tips to Comply

Make sure users can know what language the page is in! This can be marked in the html tag itself, or in any other element using the lang attribute. Specific to coding, you can also mark languages inside of code blocks!

Page Control

Specific Description

WCAG 2.0A Criterion 1.4.2: Any audio must be able to be paused, stopped, or muted.
WCAG 2.0A Criterion 1.4.2: If there any time limits on a website, users have the ability to turn it off, adjust it, extend it.
WCAG 2.0A Criterion 1.4.2: If there is content that blinks, scrolls, moves, users must have the ability to pause, stop, or hide it.

Tips to Comply

If you want students to access a website, make sure there aren't temporal hurdles or barriers that could impede students, and let them control how content moves. Let them pause carousels and audio easily, and let them take their time or change the amount of time it takes to do something. I think most of us have dealt with things that are annoyingly timed; now imagine trying to handle that same task from a keyboard. Give users control over content that auto-plays, simple as.

Semantic HTML

Specific Description

WCAG 2.0A Criterion 1.3.1: Use proper HTML markup techniques to structure your website's content.
WCAG 2.0AA Criterion 2.4.6: Headings and programmatic labels must be clear and descriptive. They do not need to be lengthy.
WCAG 2.0A Criterion 4.1.1: Make sure HTML code is clean and free of errors, particularly missing bracket closes. Also, make sure all HTML elements are properly nested.

Tips to Comply

Use semantic html anytime you write some digital content in HTML! div and span elements tell users nothing about the purpose or role of content, while there are many semantic tags that do! If you have digital content, consider how easy it is to tell what each element of a page does (or is) based solely on the tags surrounding them. Additionally, keep semantic names clear and concise for easy of navigation.

Website Navigation

Specific Description

WCAG 2.0A Criterion 2.4.2: Each page of a website needs to have a unique and descriptive page title.
WCAG 2.0AA Criterion 2.4.5: There are multiple ways to access different pages/information on a website (e.g., search bar, nav menus, sitemap, breadcrumbs, etc.).
WCAG 2.0AA Criterion 3.2.3: Keep navigation links / layout consistent throughout all pages of the website (i.e., same links in the same order).

Tips to Comply

These three criterion are about making sure that users can easily get from one page to another, and understand what page they are on relative to others at any given time. Most of this is solved by having clear navigational elements, a coherent (if possible) sitemap, and making sure your pages have titles.
